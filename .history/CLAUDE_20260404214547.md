==================================================

                              → PAID_PENDING_STOCK

  product_variants.stock field does NOT exist.
  All stock checks use: COUNT(inventory_items WHERE product_variant_id = ? AND status = 'AVAILABLE')

14. ASYNC + TRANSACTION
==================================================

- Use @Async
- Use ThreadPoolTaskExecutor (configured with corePoolSize, maxPoolSize, queueCapacity)

IMPORTANT — SELF-INVOCATION RULE:
- @Async calls MUST come from outside the declaring class (proxy must be invoked)
- Do NOT call @Async methods from within the same class

IMPORTANT — @RETRYABLE + @TRANSACTIONAL PATTERN (CRITICAL):
- NEVER place @Retryable and @Transactional on the same method in the same class
- CORRECT pattern:

  Class A (RetryableWrapper):
    @Retryable(retryFor = PessimisticLockingFailureException.class, maxAttempts = 3)
    public void allocateWithRetry(Long orderId) {
        allocationService.allocateInventory(orderId);  // calls Class B
    }

  Class B (AllocationService):
    @Transactional
    public void allocateInventory(Long orderId) { ... }

- This ensures each retry starts a FRESH transaction.
  If @Retryable wraps @Transactional in the same class, the retry runs in
  an already-aborted PostgreSQL transaction context and always fails.

17. INVENTORY RULE (CRITICAL)
==================================================

- MUST use SELECT FOR UPDATE (pessimistic locking)
- MUST sort variant IDs before locking (deadlock prevention)
- DEFAULT strategy: batch lock (single query, all variants in one transaction)

BATCH LOCK QUERY (DEFAULT — use this):
  SELECT * FROM inventory_items
  WHERE product_variant_id IN (...)
  AND status = 'AVAILABLE'
  ORDER BY product_variant_id ASC
  FOR UPDATE

ORDER for sort: product_variant_id ASC

Stock validation (cart + order):
  Use: COUNT(*) FROM inventory_items
       WHERE product_variant_id = ? AND status = 'AVAILABLE'
  There is NO stock field on product_variants.

==================================================
18. DEADLOCK PREVENTION
==================================================

- Always lock inventory_items in product_variant_id ASC order
- Use batch lock (single SELECT FOR UPDATE) instead of per-item loop locks
- Retry max 3 times on PessimisticLockingFailureException
- Use @Retryable wrapper class (see §14 for correct pattern)
- Transaction isolation: READ COMMITTED
- Keep transactions as short as possible

19. PAYMENT RULE
==================================================

- transaction_code: UNIQUE constraint on payments table
- One order can have MULTIPLE payment records (1-N relationship)
- Only ONE payment per order may ever reach status = SUCCESS
  DB-LEVEL ENFORCEMENT: partial unique index ON payments(order_id) WHERE status = 'SUCCESS'
  This makes it physically impossible to have two SUCCESS rows for the same order.
  Application-layer check alone is NOT sufficient (race condition).
- FAILED payments are kept as audit records
- Webhook MUST NOT block (fire @Async then return 200 OK immediately)

IDEMPOTENCY LOGIC (CORRECT PATTERN):

  Step 1: Receive webhook with transaction_code
  Step 2: Validate HMAC-SHA256 signature (see below)

  Step 3: SELECT payment WHERE transaction_code = ? FOR UPDATE
          ← FOR UPDATE is MANDATORY. Serializes concurrent Case B resumption.
            If two webhooks arrive simultaneously with the same transaction_code,
            only one proceeds; the other waits, then sees status=SUCCESS and exits.

  Step 3a: If payment EXISTS AND status = 'SUCCESS':
           → Return 200 OK (already fully processed)

  Step 3b: If payment EXISTS AND status = 'PENDING':
           → A previous call inserted the row but crashed before completing
           → Resume from Step 5 (do NOT re-insert)

  Step 3c: If payment NOT EXISTS:
           → Proceed to Step 4

  Step 4: INSERT payment (status = 'PENDING')
          If DataIntegrityViolationException → another thread just inserted → return 200 OK

  Step 5: Validate amount
          SELECT order WHERE id = ? (read inside the same transaction)
          IF order.total_amount != payment.amount → UPDATE payment status='FAILED', return 200 OK

  Step 6: CONDITIONAL ORDER UPDATE
          UPDATE orders SET status='PAID', updated_at=NOW()
          WHERE id = ? AND status = 'PENDING'
          ← Mandatory conditional WHERE. Guards against concurrent cron EXPIRED transition.

          IF rows_updated = 0:
            Re-read current order status:
            IF order.status = 'PAID':
              → Another concurrent thread already set this order to PAID
              → Do NOT touch payment status (it was already set to SUCCESS by that thread)
              → Return 200 OK
            IF order.status = 'EXPIRED':
              → Cron beat us. Money received, order expired. Manual refund required.
              → UPDATE payment SET status = 'FAILED'
              → Log WARNING: order_code, transaction_code, amount
              → Return 200 OK
            IF order.status = anything else (PROCESSING, COMPLETED, etc.):
              → Order was already delivered (double webhook). Do nothing.
              → Return 200 OK

          IF rows_updated = 1:
            → Continue to Step 7

  Step 7: UPDATE payment SET status = 'SUCCESS'

  (Steps 3–7 inside single @Transactional in PaymentService)

  Step 8: Call @Async allocateInventory(orderId) — OUTSIDE @Transactional scope
          ← Must be called after commit so PAID status is visible to async thread

  Step 9: Return 200 OK immediately

SIGNATURE VALIDATION (CRITICAL — must implement before production):
  - Validate webhook signature using HMAC-SHA256
  - Secret key stored in environment variable: PAYMENT_WEBHOOK_SECRET
  - NEVER hardcode the secret
  - Signed fields must match gateway documentation (VNPAY/MoMo spec)
  - Reject any webhook with invalid/missing signature with 400 BAD REQUEST

26. RECOVERY MECHANISM FOR STUCK ORDERS
==================================================

PROBLEM:
  If the application crashes after order.status = PAID is committed but before
  the @Async allocateInventory() task completes (or even starts),
  the order remains permanently in PAID status with no inventory allocated.

SOLUTION — Recovery Cron Job:
  Schedule: every 10 minutes
  Query: SELECT id FROM orders
         WHERE status = 'PAID'
         AND updated_at < NOW() - INTERVAL '10 minutes'
         FOR UPDATE SKIP LOCKED
         ← SKIP LOCKED is MANDATORY. Prevents concurrent recovery jobs (scheduled + manual
           trigger running simultaneously) from picking up the same orders and double-processing.
           Any order already locked by another recovery thread is simply skipped.

  For each stuck order:
  → Call allocateInventory(orderId) synchronously (NOT async, this is recovery)
  → Log WARNING: "Recovering stuck order {order_code}"

IMPLEMENTATION RULE:
  - allocateInventory() must be idempotent:
    Before allocating, check if a Delivery already exists for each order_item.
    If delivery exists → skip that item (already delivered in a previous run).
  - The @Transactional boundary ensures partial progress is safe.

CRON ENDPOINT (Admin):
  POST /api/v1/admin/orders/recover-stuck    (manual trigger, admin only)

27. ORDER EXPIRY CRON
==================================================

Schedule: every 5 minutes
Query: SELECT * FROM orders
       WHERE status = 'PENDING'
       AND created_at < NOW() - INTERVAL '15 minutes'
Update: UPDATE orders SET status = 'EXPIRED', updated_at = NOW()
        WHERE id = ? AND status = 'PENDING'   ← MUST use conditional WHERE

IMPORTANT:
  Use conditional WHERE status = 'PENDING' to prevent the cron from
  overwriting a PAID status set by a concurrent webhook.
  If UPDATE affects 0 rows → order was already transitioned → skip silently.
